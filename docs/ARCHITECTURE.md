## Overview

This NixOS configuration uses a modular, declarative approach built on:
- **NixOS Flakes** for reproducible builds and dependency management
- **flake-parts** for organizing flake outputs
- **import-tree** for automatic module discovery
- **Home Manager** for user-level configuration
- **Custom module system** with enable options for easy per-host customization

## Design Principles

### 1. Modularity
Every feature is self-contained in its own module with a clear `enable` option. This allows:
- Easy feature toggling per host
- Reduced cognitive load (only activate what you need)
- Clear separation of concerns

### 2. Host Separation
Different machines have different needs. Common configuration lives in `hosts/common/`, host-specific in `hosts/{host}/`.

### 3. Declarative Everything
From boot configuration to application settings, everything is declared in Nix. This ensures:
- Reproducibility across machines
- Easy rollback to previous generations
- Configuration as documentation

### 4. Security by Default
- Secure Boot enabled via Limine
- Minimal exposed services

## Module System

### Module Pattern

All modules in this configuration use the **flake-parts nixosModules pattern**. Each `.nix` file is discovered by `import-tree` and contributes to the flake via the `_:` argument:

```nix
# modules/applications/myapp/default.nix
_: {
  flake.nixosModules.myUniqueModuleName =
    { lib, config, pkgs, ... }:
    let
      cfg = config.programs.myapp;
    in
    {
      options.programs.myapp = {
        enable = lib.mkEnableOption "Enable MyApp";

        optionalPackages = lib.mkOption {
          type = lib.types.listOf lib.types.package;
          default = [ ];
          description = "Additional packages to install";
        };
      };

      config = lib.mkIf cfg.enable {
        environment.systemPackages = [ pkgs.myapp ] ++ cfg.optionalPackages;
      };
    };
}
```

The outer `_: { flake.nixosModules.<name> = ...; }` is the flake-parts module wrapper. The inner function is the standard NixOS module.

### Module Name Convention

Module names in `flake.nixosModules` are derived from their path to ensure uniqueness:
- `core/nix/btrfs.nix` ‚Üí `coreNixBtrfs`
- `modules/applications/gaming/default.nix` ‚Üí `modulesApplicationsGaming`
- `hosts/Desktop/modules/programs.nix` ‚Üí `hostDesktopModulesPrograms`

### Module Categories

1. **Application Modules** (`modules/applications/`)
   - Install and configure user applications
   - Typically add packages to `environment.systemPackages` or via Home Manager

2. **Desktop Modules** (`modules/desktop/`)
   - Configure the desktop environment
   - Handle WM, theming, login managers

3. **Hardware Modules** (`modules/hardware/`)
   - Define hardware options (GPU brand, peripherals)
   - Actual driver configuration lives in `core/hardware/`

4. **Security Modules** (`modules/security/`)
   - Authentication, encryption, GPG configuration

5. **Service Modules** (`modules/services/`)
   - User-level services (Bluetooth, Flatpak, etc.)

6. **Core Modules** (`core/`)
   - Always-on system configuration (boot, hardware drivers, Nix settings)
   - Not gated behind enable options

## Configuration Flow

### Build Process

1. **flake.nix** is evaluated via `flake-parts`
2. `import-tree` discovers all `.nix` files in `assets/`, `core/`, `hosts/`, `modules/`, `users/`
3. Each file contributes `flake.nixosModules.<name>` entries
4. Host-specific `default.nix` assembles the final system by composing modules:
   ```nix
   # hosts/{host}/default.nix
   { self, inputs, ... }:
   {
     flake.nixosConfigurations.{Host} = inputs.nixpkgs.lib.nixosSystem {
       modules = [
         self.nixosModules.assets
         self.nixosModules.usersMyUser
         self.nixosModules.core
         self.nixosModules.modules
         self.nixosModules.hostCommon
         self.nixosModules.host{Host}Hardware
         self.nixosModules.host{Host}HardwareSpecific
         self.nixosModules.host{Host}Modules
       ];
     };
   }
   ```
5. NixOS evaluates all modules, applying `enable` flags
6. Home Manager processes user-level configuration
7. System is built and can be switched to

### Option Resolution

NixOS merges all module options. Lists are merged, last scalar definition wins (use `mkForce` / `mkDefault` to control precedence):

```nix
# In modules/applications/dev/default.nix
environment.systemPackages = [ pkgs.vscode ];

# In hosts/{host}/modules/programs.nix
programs.dev.optionalPackages = [ pkgs.some-tool ];

# Result: both are installed
```

## Adding New Features

### Adding a New Application Module

1. **Create the module file:**
   ```bash
   mkdir -p modules/applications/myapp
   # create modules/applications/myapp/default.nix
   ```

2. **Write the module:**
   ```nix
   _: {
     flake.nixosModules.modulesApplicationsMyapp =
       { lib, config, pkgs, ... }:
       {
         options.programs.myapp.enable = lib.mkEnableOption "Enable MyApp";

         config = lib.mkIf config.programs.myapp.enable {
           environment.systemPackages = [ pkgs.myapp ];

           # Optional: Home Manager configuration
           home-manager.users.${config.userOptions.username} = {
             programs.myapp.enable = true;
           };
         };
       };
   }
   ```

3. **No manual import needed** ‚Äî `import-tree` discovers the file automatically.

4. **Enable in host config:**
   In `hosts/{host}/modules/programs.nix`:
   ```nix
   programs.myapp.enable = true;
   ```

### Adding a New Host

1. **Create host directory:**
   ```bash
   cp -r hosts/{existing-host} hosts/{NewHost}
   ```

2. **Generate hardware config:**
   ```bash
   nixos-generate-config --show-hardware-config > hosts/{NewHost}/hardware-configuration.nix
   ```

3. **Update `hosts/{NewHost}/default.nix`** to reference the correct nixosModule names for the new host.

4. **Customize configuration:**
   - Edit `hosts/{NewHost}/modules/programs.nix` to enable desired features
   - Update `hosts/{NewHost}/modules/userOptions.nix` for theme and user settings
   - Modify `hosts/{NewHost}/modules/hardware.nix` for hardware support

5. **Build:**
   ```bash
   sudo nixos-rebuild switch --flake .#{NewHost}
   ```

## Key Subsystems

### Boot (Limine)

- Configuration in `core/boot/boot.nix`
- Uses `limine` as the bootloader with Secure Boot support
- Dual-boot entries can be added via `boot.loader.limine.extraEntries`

### Styling System (Stylix)

- Unified theming across all applications
- Color scheme set via `config.userOptions.colorScheme`
- Wallpaper set via `config.userOptions.wallpaper` (files in `assets/wallpapers/`)
- Automatically applies to terminal, editors, desktop environment

### Hyprland Desktop Environment

- Entry point: `modules/desktop/hypr/default.nix`
- Session management via UWSM (`programs.uwsm.enable = true`)
- Includes hypridle, hyprlock, hyprpanel, hyprpaper
- Display configuration in `hosts/{host}/modules/monitors.nix`

### User Options

Each host defines a `userOptions` attribute set in `modules/userOptions.nix`:

```nix
config.userOptions = {
  browser = "firefox";           # Browser CLI command
  colorScheme = "your-scheme";   # base16 scheme name
  dots = "/home/${config.userOptions.username}/dotfiles";
  hostName = "MyHost";
  username = "myuser";
  wallpaper = "wallpaper.png";   # From assets/wallpapers/
};
```

### Hardware Configuration

```nix
config.hardwareModule = {
  gpu = {
    enable = true;
    brand = "nvidia";  # or "intel", "amd"
  };
  razer.enable = false;
};
```

GPU-specific configuration is applied based on `brand` in `core/hardware/graphics/default.nix`.

## Best Practices

### Do's
- Keep modules focused on a single feature
- Use `lib.mkEnableOption` for all optional features
- Use the `_: { flake.nixosModules.<name> = ...; }` wrapper for every module file
- Test changes with `nixos-rebuild test` before switching
- Use unique, path-derived names for `flake.nixosModules` entries

### Don'ts
- Don't mix system and user configuration in one module
- Don't enable services globally if only one host needs them
- Don't hardcode paths ‚Äî use `config.userOptions.*` variables
- Don't use imperative commands (prefer declarative config)
- Don't forget the flake-parts wrapper ‚Äî plain NixOS modules won't be discovered

## Performance Considerations

- **Build times**: Modular approach allows Nix to cache unchanged modules
- **Flake inputs**: All inputs follow `nixpkgs` to reduce cache misses
- **Garbage collection**: Automated cleanup configured in `core/nix/gc.nix`
- **Btrfs**: Weekly scrub, balance, and trim automated via `core/nix/btrfs.nix`

## Troubleshooting

### Module Not Loading
1. Check if `import-tree` can see the file (`.nix` extension, correct directory)
2. Verify the `flake.nixosModules.<name>` entry has the right name
3. Confirm the module is referenced in the host's `default.nix` module list
4. Verify `enable` option is set to `true` in host modules
5. Run `sudo nixos-rebuild build --flake .#{host}` to see detailed errors

### Option Conflicts
If you get "option defined multiple times" errors:
1. Use `lib.mkDefault` for default values
2. Use `lib.mkForce` to override
3. Check if a module is imported multiple times

### Rebuild Failures
```bash
# Test without switching
sudo nixos-rebuild test --flake .#{host}

# Check what will be built
nix flake show

# Validate flake
nix flake check
```
